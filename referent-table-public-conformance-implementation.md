# AP2認可内容とx402/EVM決済結果を照合する公開実装の指示対象対応表

- 作成日: 2026-07-21
- 出典要件: AP2–x402公開適合試験の実装・再現性・fail-closed要件
- 用途: 公開する型、関数、fixture、報告書の指示対象を実装より先に固定する
- 初版規則: `候補語` と `初出定義` は空欄で保存し、SHA-256を記録した後に確定する

| 出典 | 目的 | 具体対象 | 役割 | 前後関係 | 候補語 | 初出定義 |
| --- | --- | --- | --- | --- | --- | --- |
| AP2–x402公開適合試験の実装要件 | AP2で認可された支払いとx402/EVMで実行された支払いの一致を第三者が再現判定できるようにする | 一つの試験で照合するAP2認可、x402要求・payload、EIP-3009認可、決済結果、期待判定 | 目的 | 公開境界を固定する → 入力を検証する → 各値を照合する → 判定を記録する | 適合試験入力 (`ConformanceCase`) | 適合試験入力とは、一回の照合で使うAP2認可、x402要求・payload、EIP-3009認可、決済結果、期待判定を一つに束ねたJSON記録を指す。 |
| 公開fixtureの再現性要件 | fixtureの説明文や期待結果ではなく、固定した仕様版と検証器へ渡した値そのものを第三者が一意に照合できるようにする | AP2 commit・x402 commit/package版、`caseVersion`、`nowEpochSeconds`、AP2認可記録、x402決済記録をJCS正規化してSHA-256した43文字base64url値 | 記録 | 出典版とfixture入力を確定する → 正規化hashを計算する → 検証時に再計算する → 不一致なら判定を失敗させる | 試験入力hash (`inputHash`) | 試験入力hashとは、各fixtureに記録した固定出典版と、説明、case ID、期待判定を除く検証入力をJCS正規化してSHA-256し、base64urlで記録した非規範のfixture完全性値を指す。AP2のmandate参照には使わない。 |
| 公開検証器のmutation testと安定出力要件 | 判定ロジックの変異だけをmutation scoreで測り、公開するfailure code・path・message・computed値の一括変化は別の固定回帰で検出する | 80件のfixtureを検証した`BundleVerificationReport`全体のJCS/SHA-256値と、Strykerの文字列以外のbehavioral mutant | 記録 | 固定fixtureを全件検証する → 報告全体を正規化hashへする → 期待hashと照合する → StringLiteral以外のmutantを70%以上killする | 判定報告hash (`bundleReportHash`) | 判定報告hashとは、固定した80件への検証器出力全体をJCS正規化してSHA-256した非規範の回帰値を指す。公開failure文字列の変化を検出し、mutation scoreは条件・演算・制御フローなどのbehavioral mutantへ集中させる。 |
| AP2 v0.2 `open_payment_mandate.json`、`payment_mandate.json`、`payment_receipt.json` | ユーザーまたはエージェントが承認した受取人、金額、支払い手段、有効期限、checkout参照、mandate参照を自己申告値へ置き換えず検証する | Open Mandate SD-JWT、closed Payment Mandate KB-SD-JWT、Payment Receipt JWT、root・receipt公開JWK、検証時刻、audience、nonce、解決済みclaims、閉じたmandate参照 | 記録 | AP2署名・key binding・期限・constraint・receipt署名を実tokenで検証する → 解決済みclaimsと閉じたmandate参照を得る → x402側と照合する | AP2認可記録 (`Ap2AuthorizationRecord`) | AP2認可記録とは、署名検証に使うOpen Mandateからclosed Payment MandateまでのAP2 token chain、Payment Receipt JWT、信頼する公開JWK、検証条件、tokenから解決したclaimsを束ねた入力を指す。 |
| AP2 pinned `PaymentMandateChain.verify(expected_transaction_id, expected_open_checkout_hash)` | Open Mandateのpayment.referenceだけを確認してclosed Mandateが別Checkoutを指す状態を成功扱いにしない | 検証条件のOpen Checkout参照、署名済みOpen Mandateの`payment.reference.conditional_transaction_id`、署名済みclosed Mandateの`transaction_id` | 条件 | Open Mandate参照を検証する → closed Mandateのtransaction IDを同じ参照へ照合する → 一致後にx402へ結ぶ | 閉じたmandateのCheckout参照不一致 (`AP2_CLOSED_TRANSACTION_ID_MISMATCH` / `CLOSED_TRANSACTION_ID_MISMATCH`) | 閉じたmandateのCheckout参照不一致とは、署名検証済みclosed Payment Mandateの`transaction_id`が、検証条件として固定したOpen Checkout参照と一致しない状態を指す。前者は適合判定、後者は低レベル暗号検証APIの失敗コードである。 |
| SD-JWT object/property disclosure processing、適合試験bundle、AP2 risk data、x402 payload extensions | claim名やJSON keyをJavaScript objectのprototype操作として解釈させず、署名・入力された名前と検証対象名の対応を維持する | bundleから末端値までのraw own key、inline object key、property disclosure名の`__proto__`、`prototype`、`constructor` | 条件 | JSON又はdisclosureを復号する → `findUnsafeObjectKeyPath`でZod変換前のraw own keyを再帰検査する → 危険keyなら拒否する → 安全な入力だけを意味検証する | 危険なobject keyの経路 (`findUnsafeObjectKeyPath`) / 危険なobject key (`unsafeObjectKey`) | 危険なobject keyの経路とは、入力rootから最初に見つかった予約名までの位置を指す。危険なobject keyとは、prototype又はconstructorの意味を変えて入力された名前と読取対象名を混同させ得る予約名を指す。 |
| CAIP-2 EVM network input validation | 形式不正なchain IDを数値変換例外にせず、通常のschema不一致としてfail-closedで返す | `eip155:<positive decimal safe integer>`形式のnetwork文字列 | 条件 | 文字列形式を検査する → decimal部分がある場合だけ上限を検査する → 不正なら`INPUT_SCHEMA_INVALID`へ変換する | 安全なEVM network (`caip2EvmNetwork`) | 安全なEVM networkとは、正の10進chain IDがJavaScript safe integer内に収まるCAIP-2文字列を指す。 |
| AP2 reference、claims hash、fixture input hash validation | 43文字でも非標準表現又は32 byteへ復号できない値をnonce導出へ渡さず、schema不一致としてfail-closedで返す | paddingなしbase64urlで表したSHA-256 32-byte値 | 条件 | 文字集合と長さを検査する → 32 byteへ復号する → 同じ文字列へ再符号化できることを検査する → 不正なら`INPUT_SCHEMA_INVALID`へ変換する | 標準base64url SHA-256 (`base64UrlSha256`) | 標準base64url SHA-256とは、32 byteをpaddingなしbase64urlで一意に表し、復号後の再符号化が入力と一致する43文字列を指す。 |
| x402 Foundation `@x402/core` v2.19.0 | AP2のmerchant・ISO minor-unit金額からx402の受取address・token atomic-unit金額への対応を署名済み拡張内で固定し、固定版が出力する支払い要求、resource付き署名payload、標準決済応答をそのwire形のまま検査する | AP2 payee ID・payment_amountの複製、scheme、CAIP-2 network、asset、atomic amount、payTo、resource、accepted、payer、transaction、settlement network、任意のsettlement amount | 記録 | AP2公式claimsと署名済みx402拡張の対応を確認する → 固定版x402 clientでpayloadを作る → payloadが同じ要求を選んだか確認する → 標準決済応答が返す項目だけを照合する | x402決済記録 (`X402PaymentRecord`) | x402決済記録とは、AP2のmerchant・commerce amountとの対応、固定版が生成したPaymentRequirementsとresource付きPaymentPayload、標準SettleResponseを束ねた入力を指す。標準exact SettleResponseが返さないamountやAP2参照を必須の決済echoとして合成しない。 |
| x402 Foundation `x402Client` / `ExactEvmScheme` v2.19.0 と client hook API | 固定版producerのwire形を保ったまま、ランダムnonceをAP2の閉じたmandate参照へ決定的に結び直し、変更後messageへ再署名する | raw入力の単一exact/EIP-3009 PaymentRequirements、要求内AP2参照・nonce導出方法、EVM network、EIP-712 domain、resource、raw標準PaymentPayload、nonce・validAfter・validBefore・signature | イベント | upstream schemaが未知fieldを除去する前のraw要求を一件のstrict profileとして検査する → exact・EIP-3009・安全なEVM network・domain・検証済みAP2参照・固定導出方法を確認する → 検査済み要求で固定版clientの入力を再構築して標準payloadを生成する → after-payment-creation hookで同じAP2参照と期限を入れる → 同じEIP-712 domainで再署名する → raw producer出力を公開profile schemaでstrict検査してから固定版schemaでも検証する | AP2参照結合x402 payload生成 (`createAp2BoundX402PaymentPayload`) | AP2参照結合x402 payload生成とは、upstream変換前のraw要求を単一のexact/EIP-3009要求としてstrict検査し、検証済みAP2参照との一致と固定nonce導出方法を確認してから固定版x402 clientとExactEvmSchemeへ渡し、その公開hook境界でEIP-3009 nonceと期限を同じAP2検証結果へ置き換えて再署名し、raw出力を公開profileでstrict検査したresource付き標準PaymentPayloadを返す処理を指す。 |
| x402 Foundation `@x402/evm` v2.19.0 とEIP-3009 | 署名されたtransferWithAuthorizationが要求と認可の値を変えていないことを検査する | from、to、value、validAfter、validBefore、AP2の閉じたmandate参照をbase64url decodeしたnonce、EIP-712 domain、ECDSA署名と復元address | 手段 | 閉じたmandate参照をnonceへ変換する → EIP-712 domainと署名からaddressを復元する → 受取人・金額・期間を照合する → 決済結果へ結ぶ | EIP-3009署名記録 (`Eip3009AuthorizationRecord`) | EIP-3009署名記録とは、EIP-712 domain、transferWithAuthorizationのmessage、署名、期待するECDSA署名addressを束ねた入力を指す。EOAであること自体やsmart account署名の有効性はオフラインでは判定しない。 |
| 公開検証器のfail-closed要件とAP2 issue #299 | 欠落、未知拡張、fallback置換、同一ID・異種type、不一致を成功扱いにしない | 安定した失敗code、対象path、期待値、実値、検査順、最終判定 | 状態 | 入力不正を先に記録する → 照合不一致を記録する → 一件でもあれば失敗とする | 照合失敗記録 (`ConformanceFailure`) | 照合失敗記録とは、不一致または検証不能を安定したcode、対象path、期待値、実値とともに返す一件の判定を指す。 |
| 公開検証器の非商用・オフライン境界 | 公開検証器が秘密鍵、RPC、送金、顧客情報、商用運用を必要としないことを保証する | ローカルJSON fixtureだけを読むCLI/libraryと、検証対象外を明示する保証範囲 | 開始条件 | 公開対象から商用機能を除外する → clean cloneで検証する → 外部レビューへ渡す | オフライン検証境界 (`OfflineVerificationBoundary`) | オフライン検証境界とは、検証器がローカルJSON以外を読まず、秘密鍵、RPC、送金、顧客情報、商用運用を対象に含めない条件を指す。 |
